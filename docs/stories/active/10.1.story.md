### Feature Gate Stub — Exact Implementation

@architect (Aria) defined this interface (2026-03-02 session):

```javascript
// .aios-core/core/synapse/license/feature-gate.js
// Epic 1 MVP stub — all features enabled. Replace in Epic 2.

const FeatureGate = {
  require: (featureId, name) => {
    // Epic 1: no-op. Epic 2 replaces this with real gate check.
    // Contract: void return, NEVER throws.
  },
  isEnabled: (featureId) => true,
  reset: () => {}  // test cleanup hook
};

// Export both as default AND as named property.
// synapse-memory-provider.js uses: const { featureGate } = require('../license/feature-gate')
// Other callers may use: const featureGate = require('../license/feature-gate')
module.exports = FeatureGate;
module.exports.featureGate = FeatureGate;
```

**Why dual export:** `synapse-memory-provider.js` (Story 10.4) destructures `{ featureGate }` from this module. Both import patterns must work.

**Why this interface:**
- `require()` is called as a guard — callers expect `void` return; if feature disabled, real implementation would throw
- `isEnabled()` is used for conditional logic in callers
- `reset()` is a test utility to reset internal state (stub has no state, but interface must be present)
- [Source: @architect gate decision, PRD-SYNAPSE-001 v0.2, Section 4 Resolved Technical Decisions]

### yaml → js-yaml Migration Pattern (apply in Stories 10.2–10.5)

The MIS files on OneDrive use `require('yaml')` (the `yaml` npm package). The AIOS repo uses `js-yaml` (already present in `package.json`). When migrating any MIS file:

```javascript
// BEFORE (from OneDrive):
const yaml = require('yaml');

// AFTER (in repo):
const yaml = require('js-yaml');
```

API difference:
- `yaml.parse(str)` → `yaml.load(str)` (js-yaml)
- `yaml.stringify(obj)` → `yaml.dump(obj)` (js-yaml)

Check each migrated file for these API calls and update accordingly.
[Source: PRD-SYNAPSE-001 v0.2, Resolved Technical Decisions]
