## Dev Notes

### Class F Definition

Class F = "victim suites" ÔÇö test files that pass in isolation but fail non-deterministically when Jest runs the full suite in parallel. The contamination source is external to the victim suite itself: another suite (the "contaminator") leaves shared state dirty, which the victim then encounters when scheduled to the same Jest worker process.

> Source: `docs/plans/flaky-tests-diagnostic.md` ÔÇö Victim classification section.

### Why Classes AÔÇôE Fixes Did Not Resolve These 7

Stories 11.3 (Class A), 11.4 (Class B), and 11.5 (Class C/D/E) targeted specific contamination sources. However, after those fixes, these 7 suites still fail in parallel ÔÇö meaning either:
1. A contaminator was not fully fixed (partial fix in 11.3ÔÇô11.5), OR
2. A new contamination vector was revealed once the original sources were eliminated, OR
3. These 7 suites have an internal interaction when scheduled together on the same worker

### Contamination Hypotheses (Priority Order)

**Hypothesis 1 (HIGH) ÔÇö `jest.spyOn(process, 'exit')` not restored on test timeout**

`config-cli.test.js` spies on `process.exit` in `beforeEach` and restores in `afterEach`. If a test times out before `afterEach` runs, the spy leaks into subsequent tests in the same worker. This can cause other suites' legitimate `process.exit` calls to throw `ProcessExitError` unexpectedly.

**Investigation:** Run `npx jest tests/config/config-cli.test.js --testTimeout=1` to force timeouts and observe if spy leaks. Check if `afterEach` in config-cli explicitly uses `try/finally` to guarantee restore.

**Fix:** Wrap spy restore in `try/finally` block or use `jest.restoreAllMocks()` in a global `afterEach`.

---

**Hypothesis 2 (HIGH) ÔÇö `jest.clearAllMocks()` in `greeting-builder.test.js` leaves factory mock stale**

`greeting-builder.test.js` uses `jest.clearAllMocks()` which resets call counts but NOT the `greeting-preference-manager` factory mock set at module level:
```js
jest.mock('...greeting-preference-manager', () => {
  return jest.fn().mockImplementation(() => ({ ... }));
});
```
The factory is set once and never refreshed. If another suite runs in the same worker and `GreetingPreferenceManager` is required (not mocked), the factory override may persist in unexpected ways.

**Fix:** Replace `jest.clearAllMocks()` with `jest.resetAllMocks()` and ensure the `greeting-preference-manager` factory is re-applied explicitly in `beforeEach`.

---

**Hypothesis 3 (MEDIUM) ÔÇö `self-learner.test.js` secondary contamination after Class A fix**

Story 11.3 fixed the `mkdtempSync` for self-learner. But self-learner has 1000+ lines and complex `beforeEach`/`afterEach` at nested `describe` levels. If an inner `afterEach` throws (e.g., `ENOENT` on cleanup of a file that failed to create), Jest may skip subsequent cleanup, leaving partial state.

**Investigation:** Run `npx jest tests/core/synapse/memory/self-learner.test.js --verbose` 5├Ś and look for any `afterEach` errors in output.

**Fix:** Wrap all `afterEach` cleanup in `try/catch` to prevent cleanup failures from surfacing as test failures.

---

**Hypothesis 4 (MEDIUM) ÔÇö Shared `execution-test-helpers.js` module state**

`semantic-merge-engine.test.js` and `wave-executor.test.js` both import from `tests/core/execution-test-helpers.js`. If this helper module holds any mutable state at module level (not just factory functions), and two Jest workers import it concurrently via the same Node.js process, there could be contamination.

**Investigation:** Check `execution-test-helpers.js` for any mutable module-level variables.

**Fix:** Ensure all state in `execution-test-helpers.js` is created fresh per call (no module-level mutables).

---

**Hypothesis 5 (LOW) ÔÇö `session-state.test.js` async cleanup race**

`session-state.test.js` uses `fs.promises` (async) in `beforeEach`/`afterEach`. If a test suite running in the same worker starts before the async cleanup completes (Jest timing), there could be leftover temp files. However, Jest waits for returned Promises, so this should not normally happen.

**Investigation:** Add `--detectOpenHandles` flag: `npx jest tests/core/orchestration/session-state.test.js --detectOpenHandles`

## Dev Agent Record

### Completion Notes

**Root cause:** Two separate issues found:

1. **`workflow-navigator-integration.test.js` (deferred 11.8)** ÔÇö patches `fs.readFileSync` and `fs.existsSync` directly on the `fs` module object (not via `jest.mock`). When tests fail (as expected for this deferred suite), the restore lines are never reached. The patched `fs.readFileSync` returns `yaml.dump(sessionStateData)` ÔÇö a YAML string containing `title: Activation Pipeline`. Any subsequent test file in the same Jest worker that calls `fs.readFileSync` gets YAML back instead of file content. Jest's own source formatter (`@jest/console` Ôćĺ `jest-message-util`) uses `fs.readFileSync` to read source files for error context, gets YAML, and throws `SyntaxError: Unexpected identifier 'Pipeline'`. **Fix:** wrapped all 8 occurrences of direct `fs` monkey-patching in `try/finally` blocks.

2. **`wave-executor.js` ÔÇö unreferenced timers** ÔÇö `executeTaskWithTimeout()` creates a `setTimeout` with the default 10-minute `taskTimeout`. When tasks complete quickly (as in tests), the 10-minute timer stays pending and keeps the Jest worker alive (open handle). Additionally, the `finally` block creates a 5-second cleanup timer. **Fix:** added `.unref()` to both timers so they don't prevent process exit.

**Investigation methodology:**
- 6 of the 7 Class F suites were already passing in full runs (Stories 11.3-11.5 had fixed their contamination)
- The actual contaminator was the deferred `workflow-navigator-integration.test.js` (11.8) which was leaving `fs` in a dirty state
